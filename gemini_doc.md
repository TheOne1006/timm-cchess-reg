# iOS 端侧中国象棋棋谱识别：全卷积网格预测架构方案

## 一、 架构定位与核心思想

本方案在学术界被称为 **全卷积网格预测 (Fully Convolutional Grid Prediction)** 或 **空间张量分类 (Spatial Tensor Classification)**。

**核心思想：**
将整张经过透视变换的棋盘图像作为输入，通过全卷积网络进行特征提取和空间降维，最终直接输出一个空间拓扑结构与真实棋盘完全一致的 `9 x 10` 分类矩阵。

**为什么放弃 Object Detection（如 YOLO）与 SwinTransformer？**
1. **透视形变容错率低：** 透视变换通常由前置任务预测，不可避免会存在微小扭曲。目标检测依赖精确的 Bounding Box 回归，对这种全局性的微小形变极其敏感；而基于感受野交叠的纯卷积网络具有更好的平移等变性和容错性。
2. **CoreML 导出困难与 ANE 兼容性差：** SwinV2 包含复杂的窗口注意力（Window Attention）、特征图滚动（Roll）和自定义的张量 Reshape/Pooling 操作。这些算子极易导致 CoreML 导出失败，或在 iOS 设备上退化至 CPU 运行，引发严重的发热与卡顿。

## 二、 整体网络管线 (Pipeline)

网络的整体数据流遵循“高分辨率输入 -> 纯 CNN 主干 32 倍下采样 -> 步长卷积 2 倍下采样 -> 感受野融合 -> 逐点分类”的清晰逻辑。

### 1. 输入阶段 (Input)
* **尺寸设定：** 将透视校正后的棋盘图像 Resize 为固定分辨率 **576 x 640**。
* **原因：** 象棋中文字符（如“車”、“馬”、“帥”）包含大量高频细节特征。较高的分辨率能确保这些特征在深层网络中不被过早平滑掉。图像为标准 RGB 3 通道。

### 2. 主干网络阶段 (Backbone)
* **模型选择：** 推荐使用 **ConvNeXt (Atto/Femto 版本)** 或 **MobileNetV3-Large**。
* **处理逻辑：** 移除原版网络末尾的全局平均池化层（Global Average Pooling）和全连接层（FC Layer）。
* **输出状态：** 标准 CNN 主干自带 32 倍的最大下采样率。输入 576 x 640 经过主干后，将自动输出尺寸为 **18 x 20 x C** 的特征图（C 为通道数，通常为 256 或更高）。

### 3. 核心降维阶段 (可学习的下采样)
* **组件：** 单层二维卷积 (Conv2d)。
* **参数设定：** 卷积核大小 **3x3**，步长 (Stride) **2**，填充 (Padding) **1**。
* **输出状态：** 将空间维度精确缩小一半，输出完美的 **9 x 10 x C** 特征图。
* **优势：** 这是“方案一”的核心。相比于粗暴的池化（Pooling），步长卷积是一种**可学习的下采样**。卷积核会自主学习如何将 2x2 像素区域内最重要的字形特征（如“炮”的偏旁）融合为一个最具代表性的高维特征向量，极大地增强了对透视形变的鲁棒性。

### 4. 上下文融合阶段 (Contextual Module)
* **目标：** 解决单点识别准确率瓶颈，引入相邻棋子的布局影响（如“连环马”、“当头炮”）。
* **组件：** 多分支空洞卷积模块。
* **处理逻辑：** 在 `9 x 10 x C` 的特征图上，并联执行以下操作，随后将特征相加或拼接：
  1. **标准 3x3 卷积：** 感受野覆盖周围一圈（九宫格），获取紧邻棋子的信息。
  2. **空洞卷积 (Dilation=2) 的 3x3 卷积：** 扩大感受野，覆盖“日”字对角或更远的邻居，模拟象棋的特殊走法关联。
* **优势：** 标准空洞卷积在 ANE 上拥有完美的硬件加速支持，能够在不增加特征图尺寸的前提下获取全局棋局上下文。

### 5. 分类输出阶段 (Classification Head)
* **组件：** 1x1 卷积层。
* **处理逻辑：** 将通道数 C 映射到 **16**（代表：14 种红黑棋子 + 1 种空白 + 1 种棋盘边缘/背景错误）。
* **输出状态：** 得到最终的 **9 x 10 x 16** 张量。
* **激活：** 在最后一个维度（Channel=16）上执行 Softmax 操作，输出每个网格点属于各分类的概率分布。

## 三、 CoreML 导出与 iOS 端侧优化铁律

为了确保你的模型能在 iPhone 的 Apple Neural Engine (ANE) 上实现满血的实时推理（榨干 NPU 性能），在 PyTorch 建模和导出 ONNX 时必须严格遵守以下原则：

1. **绝对禁止动态形状 (No Dynamic Shapes)：**
   * 在模型前向传播（Forward）的任何代码中，不要使用涉及动态计算维度的 `reshape(-1, ...)` 或基于输入实时计算形状的 Adaptive Pooling。
   * 确保导出时张量形状被彻底锁死为：`[Batch=1, Channel=3, Height=640, Width=576]`。
2. **通道数对齐 (Channel Alignment)：**
   * ANE 的硬件架构对通道数极其敏感。在设计你自己添加的“核心降维层”和“上下文融合层”时，**务必保证通道数（Channel）是 32 的整数倍**（如 32, 64, 128, 256）。这能避免 ANE 在内部执行低效的内存填充（Padding），让推理速度提升 30% 以上。
3. **安全激活函数 (Safe Activations)：**
   * 坚持使用最基础的激活函数：**ReLU、ReLU6 或 GELU**。
   * 避免使用 Swish、Mish 等较新或复杂的激活函数，这些极其容易导致模型在 CoreML 中被拆分，部分算子降级到 CPU 运行，造成严重的性能瓶颈。
4. **算子纯粹性：**
   * 整个网络只允许出现：`Conv2d`, `BatchNorm2d`, `ReLU/GELU`, 以及基础的 `Add` (张量相加) 或 `Concat`。保持算子的纯粹是 iOS 端侧部署成功的关键。
